# Task 0C — Isolation Test Harness

Build a small, focused tool for running AVFoundation/VideoToolbox/CIContext configurations in isolation so we can empirically map out what does and doesn't work on this M2 Pro Mac without repeatedly hanging the developer's machine. The harness exists to answer specific hypotheses generated by task-0B (research), and to bisect failure modes so task-0A (recording pipeline) can be unblocked with evidence instead of speculation.

This is a coding task, but the goal is a **measurement instrument**, not a shippable feature. The harness should favour observability, reproducibility, and safety over production quality.

**This task is only about building the harness.** The test plan that it will be run against, along with the decision framework for interpreting results, lives in `docs/tasks-todo/task-1-run-test-harness-tests.md`. Don't conflate "build the tool" with "run the experiments" — they're separate commits, separate sessions, and separate audiences.

## Context

Read `docs/m2-pro-video-pipeline-failures.md` in full before starting. It documents the four failure modes we've observed and the diagnostic evidence for each. In particular, failure mode 4 (the 1440p-preset kernel-level IOGPUFamily deadlock) is the headline problem this harness exists to help diagnose.

Also worth skimming:

- `docs/tasks-todo/task-0A-encoder-contention-and-camera-pipeline.md` — the task this harness supports. Phase 2 landed; Phase 2b partially landed and triggered failure mode 4; Phase 3 and 4 are blocked on understanding what actually works on this hardware.
- `docs/tasks-todo/task-0B-video-pipeline-research.md` — the research task running in parallel. It will produce a list of concrete hypotheses that this harness should be able to validate.

The short version of why this task exists: our mental model of what combinations of hardware video sessions are safe on M2 Pro has been wrong twice in one session, and we've hung the developer's Mac twice requiring hard power-button reboots. We need a way to test configurations that is **safer** and **more systematic** than doing it in the real app.

## Goal

Produce a standalone tool that can:

1. Spin up arbitrary combinations of video writers (composited HLS H.264, raw H.264, raw ProRes 422 Proxy, raw audio AAC) with configurable resolutions, bitrates, and per-session tuning parameters.
2. Drive them with synthetic pixel buffers (default) or real `ScreenCaptureKit` + `AVCaptureSession` capture (opt-in, for specific tests).
3. Optionally run a `CIContext` compositor stage for tests that include composition.
4. Run each configuration for a configurable duration with a hard wall-clock watchdog.
5. Detect pass/fail per run based on observable signals (GPU errors, writer status, segment cadence, output file sanity).
6. Log enough instrumentation that a stalled or failed run produces useful post-mortem data without requiring a kernel spindump.
7. Be safely scriptable so we can run a sequence of test configurations without manually driving the app for each one.

## Out of scope

- **Replacing or replicating the real app's recording pipeline.** The harness is allowed to be messier, simpler, and more directly coupled than `RecordingActor` / `CompositionActor` / `WriterActor` in the main app. It can duplicate a bit of code if that makes it cleaner.
- **Shipping.** Nothing from the harness gets shipped to users. It lives in the repo, builds locally, and is for development use.
- **Executing the test plan.** The actual experiments the harness runs are task-1's job. This task ends when the harness is working end-to-end with the Tier 1 (single-component isolation) suite producing clean pass/fail output.
- **Fixing the failure modes.** This task does not touch the main app's recording pipeline. Task-0A will do that later, using the data task-1 produces.

## What the harness IS

- A focused diagnostic tool with minimal ceremony.
- A way to vary ONE variable at a time and see what happens.
- A safety-first testing environment that protects the developer's Mac to the extent possible.
- A record-keeper that writes structured test results to disk for post-hoc analysis.
- Throwaway-adjacent: if we need to rewrite it later we can, but we'd like it to serve us for the remainder of task-0A at least.

## What the harness IS NOT

- A replacement for the production recording pipeline.
- A UI-driven interactive app.
- A performance benchmark or quality comparison tool.
- A reproduction of the exact threading model of the real app. It should mirror the important parts (concurrent writers, compositor calling CIContext, ScreenCaptureKit feeding frames) but doesn't have to faithfully reproduce everything.

## Core capabilities required

### Configurable writer set

The harness needs to be able to instantiate any combination of:

- **Composited HLS writer** — `AVAssetWriter` with `outputFileTypeProfile = .mpeg4AppleHLS`, `preferredOutputSegmentInterval = 4s`, configurable H.264 resolution and bitrate, H.264 High profile. Same configuration shape as `WriterActor.swift` in the main app.
- **Raw H.264 writer** — `AVAssetWriter` writing to a plain `.mp4`, configurable H.264 resolution and bitrate. Mirrors the `RawStreamWriter.swift` `.videoH264` case.
- **Raw ProRes writer** — `AVAssetWriter` writing to a plain `.mov`, ProRes 422 Proxy codec, configurable input resolution. Mirrors the `RawStreamWriter.swift` `.videoProRes` case.
- **Raw audio writer** — `AVAssetWriter` writing to `.m4a`, AAC-LC. Mirrors `RawStreamWriter.swift` `.audio`.

Each writer type should be independently enable/disable-able. A test that runs just "ProRes + H.264 composited at 1080p" should be one line of config.

### Configurable compositor stage (optional)

When enabled, the harness runs a CIContext-based compositor that reads one or two input CIImages and renders a composited output to a CVPixelBuffer that feeds the composited HLS writer. Configuration:

- Output resolution (independent of writer resolution, though in practice they should match)
- Whether to include a camera overlay (PiP via circle mask, same approach as `CompositionActor.compositeFrame`)
- Whether to use Lanczos scaling on the screen input
- Whether to use `render(to:bounds:colorSpace:)` (current main-app path) or `startTask(toRender:to:)` (task-0A Phase 3's proposed path)

### Frame sources

Two modes:

1. **Synthetic frames (default).** The harness generates frames at 30 fps from a pixel-buffer pool. Options:
   - Screen-like BGRA frames at configurable resolution (e.g. 3840×2160 for native Retina)
   - Camera-like 420v YCbCr frames at configurable resolution
   - Configurable content: solid colour, gradient, noise, or a moving shape (to exercise the encoder with non-static content without needing a real capture source)
   - Colour-space attachments matching real capture (sRGB for screen, Rec. 709 for camera)

2. **Real capture (opt-in).** `SCStream` for screen, `AVCaptureSession` for camera. Same approach as `ScreenCaptureManager.swift` and `CameraCaptureManager.swift` in the main app. Enabled only for specific tests that explicitly need the capture layer.

Synthetic frames are strongly preferred for most tests because they remove capture-layer variance as a confounder. If a test fails with synthetic frames, we know the capture layer is not the cause.

### Configurable per-session tuning

The whole reason this harness exists is to test tuning knobs we don't currently use. It needs to let tests set arbitrary `VTCompressionSession` properties, CVPixelBufferPool attributes, and AVAssetWriter input properties at configuration time. The specific properties to cover (driven by task-0B research) include at minimum:

- `kVTCompressionPropertyKey_RealTime`
- `kVTCompressionPropertyKey_MaxFrameDelayCount`
- `kVTCompressionPropertyKey_MaximizePowerEfficiency`
- `kVTCompressionPropertyKey_AllowFrameReordering`
- `kVTCompressionPropertyKey_AverageBitRate`
- `kVTCompressionPropertyKey_ExpectedFrameRate`
- `kVTVideoEncoderSpecification_RequireHardwareAcceleratedVideoEncoder`
- `kVTVideoEncoderSpecification_EnableHardwareAcceleratedVideoEncoder`
- `kCVPixelBufferPoolMinimumBufferCountKey`
- `kCVPixelBufferPoolMaximumBufferAgeKey`
- `SCStreamConfiguration.pixelFormat` (BGRA, 420v, 420f)
- `SCStreamConfiguration.queueDepth`
- `SCStreamConfiguration.width` / `height` (to test sub-native-resolution screen capture)
- Whether `VTCompressionSessionPrepareToEncodeFrames` is called before the first frame

This list is intentionally non-exhaustive. New properties should be easy to add as task-0B research discovers them.

### Instrumentation and logging

For every test run, the harness captures:

- **Configuration snapshot** — full config used for the run, serialised as JSON, so the test is exactly reproducible.
- **Per-writer events** — timestamped log of `startWriting`, first sample appended, segment output (for HLS), completion, any errors.
- **Per-frame cadence** — timestamp when each frame was submitted to each writer, timestamp when segments were emitted, running count of appends/drops.
- **GPU error log** — anything matching `kIOGPUCommandBufferCallback*` or CoreImage errors observed during the run. Use `log stream` as an external process or `os_log` predicates within the harness.
- **Writer final status** — each `AVAssetWriter`'s `status` and `error` after `finishWriting`.
- **Output file metadata** — sizes, durations, codec info for every file the harness produces.
- **System snapshot at start and end** — `vm_stat`, `sysctl hw.cpufrequency`, `powermetrics` one-shot output, `ioreg -c IOSurfaceRoot -l` truncated if possible. This gives us a before/after view of IOSurface and memory pressure.
- **Process-level CPU/memory sample** — sample the harness process itself during the run (e.g. every second) to see if we're growing unbounded.

Logs go to a single directory per test run: `test-runs/{timestamp}-{test-name}/`. Each run produces at minimum `config.json`, `events.jsonl`, `stdout.log`, `stderr.log`, `system-snapshot-start.txt`, `system-snapshot-end.txt`, and `result.json` (the pass/fail summary).

### Pass/fail determination

For each run, the harness computes a result:

- **PASS** if all of:
  - Test ran to its configured duration ± small grace
  - All writers ended with status `.completed`
  - No `kIOGPUCommandBufferCallback*` errors observed
  - Output files exist and have non-trivially-nonzero size
  - Segment cadence (if HLS) stayed within ±10% of the 4 s target after the first segment
- **DEGRADED** if the test completed but one or more of the above soft-fail conditions tripped (e.g. some GPU errors but recording continued, or segment cadence drifted but recovered)
- **FAIL (recorded)** if a writer failed, an error was thrown, or the output is missing/corrupt, but the harness still returned
- **FAIL (killed)** if the harness's own wall-clock watchdog fired and terminated the run

A `DEGRADED` or `FAIL (recorded)` result is expected for configurations we know are on the edge. A `FAIL (killed)` result means the test probably would have hung the Mac if left to run — which is the whole point of the watchdog.

## Safety mechanisms

These are non-negotiable. The harness will, at some point, be told to run a configuration that triggers a hang. Three defences:

### Last-known-good marker

Before starting a test run, the harness writes `test-runs/_in-progress.json` containing the test name and full config. On successful completion, it deletes this file. If the Mac hangs and reboots, the file persists. The next time the harness runs, it checks for this file first — if present, it's a signal that the last run hung and that config should be flagged as "confirmed dangerous" rather than retried blindly.

### Wall-clock watchdog

The harness runs every test with a hard wall-clock deadline equal to `duration + 10 seconds`. If the test hasn't reported completion by then, a background timer kills the process via `exit()`. This won't save us from a true kernel-level hang (the thread that needs to call `exit()` might be stuck too), but it catches the large class of "userspace stall with a stuck thread" cases before they escalate.

Ideally the watchdog fires from a **thread separated from the main recording pipeline**, so it's not itself susceptible to being blocked by the stall it's trying to abort. A pthread-based timer running at elevated priority is reasonable.

### Dry-run mode

The harness supports `--dry-run` which validates a config, prints what it WOULD do, but does not actually call `AVAssetWriter.startWriting` or any other AVFoundation entry point that could trigger the failure. This lets us sanity-check test plans before executing them for real.

Dry-run should also be the first thing any new test configuration is run with.

### Start small, step up — enforced by the test runner script

The harness itself doesn't know about the test plan — it just runs individual configurations. The **test runner** (a shell script or Swift driver that calls the harness for a sequence of configs) is responsible for running tests in order of increasing risk, and stopping at the first kill or failure so we don't repeatedly trigger the dangerous config.

The runner should:

1. Refuse to start if `_in-progress.json` exists (signalling a previous hang)
2. Run tier-1 tests first (safer configs), then tier-2, etc.
3. Stop on any `FAIL (killed)` result
4. Stop on any `FAIL (recorded)` result unless `--continue-on-fail` is passed
5. Write a summary at the end: which configs passed, which degraded, which failed, which were skipped

## Design approach

The harness should be a **new Xcode target** in the existing `app/LoomClone.xcodeproj` project, not a separate package. This avoids the TCC / code-signing / entitlement maze we'd hit with a standalone CLI tool, and lets the harness share utility code with the main app where useful (though it shouldn't depend on the main app's actors or UI layer).

**Target type:** macOS app bundle. Yes, an app. Not a CLI tool. The reason is TCC — screen capture and camera access require the containing process to be a signed, bundled app with the right entitlements, and we already have that infrastructure in place for LoomClone. A CLI tool would need to redo all of that or run in a mode that can't exercise real capture.

**Entry point:** the harness app has a minimal UI (a single window showing current test name + status, or even just a log scroll) or no UI at all (just runs and exits). It reads its configuration from a JSON file whose path is passed via command-line arguments (`open -a TestHarness --args --config /tmp/test.json`). On launch, it loads the config, runs the test, writes results to disk, and exits.

**No SwiftUI requirements.** If a minimal window is needed for TCC prompts or dock presence, AppKit is fine. We don't need this to look nice.

**File layout suggestion** (not mandatory — adjust as makes sense during implementation):

```
app/
  TestHarness/
    TestHarness.swift              // Main entry point, config loader, result writer
    HarnessConfig.swift            // Config schema
    HarnessResult.swift            // Result schema
    HarnessRunner.swift            // Orchestrates a single test run
    Writers/
      CompositedHLSWriter.swift    // Minimal analogue of WriterActor
      RawH264Writer.swift          // Minimal analogue of RawStreamWriter .videoH264
      RawProResWriter.swift        // Minimal analogue of RawStreamWriter .videoProRes
      RawAudioWriter.swift         // Minimal analogue of RawStreamWriter .audio
    Sources/
      SyntheticFrameSource.swift   // CVPixelBuffer generator
      CapturedFrameSource.swift    // Wraps SCStream + AVCaptureSession for real-capture tests
    Compositor/
      HarnessCompositor.swift      // Minimal analogue of CompositionActor
    Observability/
      EventLog.swift               // Structured events.jsonl output
      SystemSnapshot.swift         // vm_stat / ioreg / powermetrics capture
      WatchdogTimer.swift          // Wall-clock kill-switch
    Scripts/
      run-tier-1.sh                // Shell script that runs tier 1 configs in order
      test-configs/
        tier-1/
          01-prores-only.json
          02-h264-1080p-only.json
          ...
```

None of this layout is mandatory — if it makes sense to collapse Writers/ into one file for simplicity, do that. The point is separation of concerns: sources produce frames, writers consume them, the compositor is optional between them, observability captures what's happening, watchdog keeps us safe.

## Instrumentation targets

Things the harness should try to capture during a run that the real app doesn't currently log:

- **IOSurface pool state** — we don't have direct API access to IOSurface pool internals, but `ioreg -c IOSurfaceRoot -l` at start and end gives us a rough picture. Capture both.
- **GPU utilisation** — `powermetrics --samplers gpu_power --sample-count 1 -i 100 --hide-cpu-duty-cycle` gives one-shot GPU stats. Or run `powermetrics` as a background subprocess during the test and capture its output.
- **Thread count in the harness process** — `ps -M $PID` or just iterating through `pthread` internals. Useful for detecting "thread explosion" patterns.
- **Dispatch queue depth for the compositor** — if we build the compositor with a serial queue, we can check `DispatchQueue.sync` completion times.
- **AVAssetWriter internal state** — status, error, `isReadyForMoreMediaData` true/false per frame per writer. The latter tells us if we're back-pressured.
- **CMTime / PTS gaps** — if a frame gets dropped or a segment stretches, we want a record of when.
- **Metal command buffer scheduling events** — if we create our own `MTLCommandQueue` and `MTLCommandBuffer` with scheduling handlers, we can log when buffers are scheduled, committed, and completed. This is how we'd detect stalls before the GPU watchdog fires.

Not everything has to be captured — some of this may be overkill or costly. Start with event log + system snapshots + pass/fail, and add the more exotic instrumentation if we find ourselves needing it.

## Deliverable

1. **Harness source code** in `app/TestHarness/` (or a similar location in the Xcode project). New Xcode target named `LoomCloneTestHarness` or similar. Builds to a debug `.app` bundle.
2. **Initial test configuration files** checked into the repo at `app/TestHarness/Scripts/test-configs/tier-1/` covering Tier 1 (single-component isolation) as a proof the harness end-to-end works. The broader test plan — Tiers 2–5 — is the next task's job, not this one's.
3. **A runner script** at `app/TestHarness/Scripts/run-tier-1.sh`. Shell script, not Swift — it needs to persist across reboots and be trivial to run after a hang.
4. **A `README.md`** at `app/TestHarness/README.md` covering: how to build, how to run a single test, how to run a tier, how to interpret results, how to add new tests, how to recover after a hang (check for the last-known-good marker). This is the evergreen user-facing doc for the harness.
5. **Tier 1 baseline results committed** — run the Tier 1 suite at least once on the dev machine and commit a short markdown summary at `test-runs/tier-1-baseline-<date>.md` recording what passed / failed / degraded. Run directories themselves stay gitignored.

## Hard constraints

- **Do not reproduce failure mode 4 in the main LoomClone app.** The developer's Mac should never hang again because of a manual test. The harness itself is the entire safety boundary — don't add fallbacks that run tests through the main app.
- **The harness process must be killable.** If a test runs longer than its budget, the wall-clock watchdog must fire. If the user sends SIGINT, the harness must clean up and exit. If either of these fails reliably, the safety story collapses.
- **No changes to the main LoomClone recording pipeline in this task.** The harness is additive. Any findings that suggest a main-app change get written up and handed to task-0A.
- **Results go to a fixed location that survives reboots.** `test-runs/` under the project root, committed to `.gitignore` so individual run dirs aren't staged, but the directory itself is tracked with a `.gitkeep`.
