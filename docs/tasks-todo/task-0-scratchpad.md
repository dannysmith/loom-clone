In an ideal world, there'd also be an easy way for us to ensure that unlisted videos have even longer slugs, which are basically impossible to guess. Now the difficulty we have here is that all videos start as unlisted and need the user to explicitly make them Public or Private. Now we don't really need to care about private videos here. I think it's fine if private videos have the W3W form, since nobody can access these at all unless they're auth'd via the admin panel or an admin token. But I guess one option here could be to provide a pre-recording option for visibility which the user sets in the macOS app before they hit record, which dictates the type of slug that is initially generated (this could obviously be changed later on in the usual way). And I guess another option would be to always generate `contacts-jetted-dean-977e2d0f2048473dba3cea3c2227a210`-type slugs by default and then update the slig to just `contacts-jetted-dean` if the visibility is changed from unlisted ever. Well it should be clear that everything I'm talking about here applies only to the generation of random slugs - If I manuallly edit a slug Then that's the slug we want regardless of the video's visibility (there will be plenty of unlisted videos where I want a short custom URL, and in a few cases this may even apply to private videos too). So whatever we do with this, should be purely to do with slug generation and potentially with automatically changing generated slugs when the visibility is changed. It shouldn't be to do with like routing inbound traffic, so to speak. Either way, before we build this we need to do some thinking about exactly how we want this to work and the various options. Obviously this also all needs to work with the current system of redirects when slugs are changed.
